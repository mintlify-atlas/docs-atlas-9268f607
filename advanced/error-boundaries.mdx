# Error Boundaries

Error Boundaries are a native RyunixJS component that catch JavaScript errors anywhere in their child component tree, log those errors, and display a fallback UI instead of crashing the entire application.

## Why Error Boundaries Matter

Without error boundaries, a JavaScript error in any component can crash your entire application. Error boundaries provide a safety net that:

<CardGroup cols={2}>
  <Card title="Prevent Full Crashes" icon="circle-check">
    Isolate errors to specific UI sections instead of breaking the entire app
  </Card>
  <Card title="Improve UX" icon="face-smile">
    Show user-friendly fallback UI instead of blank screens
  </Card>
  <Card title="Error Reporting" icon="bug">
    Capture error details for logging and monitoring services
  </Card>
  <Card title="Graceful Degradation" icon="stairs">
    Allow other parts of the application to continue functioning
  </Card>
</CardGroup>

## Basic Usage

Wrap any component tree with an ErrorBoundary to catch errors:

```javascript
import { ErrorBoundary } from '@unsetsoft/ryunixjs'

function App() {
  return (
    <ErrorBoundary fallback={<div>Something went wrong</div>}>
      <UserProfile />
      <Dashboard />
      <ActivityFeed />
    </ErrorBoundary>
  )
}
```

If any component inside the boundary throws an error during rendering, the fallback UI is displayed instead.

## How It Works

ErrorBoundary hooks directly into the RyunixJS reconciler's `performUnitOfWork` loop:

<Steps>
  <Step title="Error Detection">
    When an error occurs during fiber reconciliation, the reconciler marks the fiber with `stateError`
  </Step>
  <Step title="Boundary Check">
    The reconciler walks up the fiber tree to find the nearest ErrorBoundary component
  </Step>
  <Step title="Fallback Render">
    The ErrorBoundary detects `state.wipFiber.stateError` and renders the fallback UI
  </Step>
  <Step title="Error Isolation">
    Child components below the boundary are unmounted and replaced with the fallback
  </Step>
</Steps>

<Note>
Error boundaries work at the reconciler level, making them more reliable than try/catch blocks which only work for imperative code.
</Note>

## Fallback Patterns

### Static Fallback

Provide a simple JSX element:

```javascript
<ErrorBoundary fallback={<div>Oops! Something went wrong.</div>}>
  <MyComponent />
</ErrorBoundary>
```

### Function Fallback

Use a function to access error details:

```javascript
<ErrorBoundary
  fallback={(error) => (
    <div>
      <h2>Error Occurred</h2>
      <p>{error.message}</p>
      <pre>{error.stack}</pre>
    </div>
  )}
>
  <MyComponent />
</ErrorBoundary>
```

The function receives the error object, allowing you to:
- Display error messages to users
- Log errors to monitoring services
- Show different UI based on error type

### Component Fallback

Create reusable error UI components:

```javascript
function ErrorFallback({ error }) {
  return (
    <div className="error-container">
      <h1>Something went wrong</h1>
      <details>
        <summary>Error details</summary>
        <pre>{error.message}</pre>
      </details>
      <button onClick={() => window.location.reload()}>
        Reload page
      </button>
    </div>
  )
}

function App() {
  return (
    <ErrorBoundary fallback={(error) => <ErrorFallback error={error} />}>
      <Dashboard />
    </ErrorBoundary>
  )
}
```

## Strategic Error Boundary Placement

### Application-Level Boundary

Wrap your entire app for top-level error handling:

```javascript
function App() {
  return (
    <ErrorBoundary fallback={<FullPageError />}>
      <Router>
        <Routes />
      </Router>
    </ErrorBoundary>
  )
}
```

### Route-Level Boundaries

Isolate errors to specific routes:

```javascript
function Routes() {
  return (
    <>
      <ErrorBoundary fallback={<PageError />}>
        <Route path="/dashboard" component={Dashboard} />
      </ErrorBoundary>
      
      <ErrorBoundary fallback={<PageError />}>
        <Route path="/profile" component={Profile} />
      </ErrorBoundary>
    </>
  )
}
```

### Component-Level Boundaries

Wrap individual complex components:

```javascript
function Dashboard() {
  return (
    <div className="dashboard">
      <Header />
      
      <ErrorBoundary fallback={<WidgetError />}>
        <StatsWidget />
      </ErrorBoundary>
      
      <ErrorBoundary fallback={<WidgetError />}>
        <ChartWidget />
      </ErrorBoundary>
      
      <Footer />
    </div>
  )
}
```

<Tip>
Use multiple granular error boundaries rather than one top-level boundary. This allows parts of your UI to continue functioning even when others fail.
</Tip>

## Error Logging and Monitoring

Integrate error boundaries with monitoring services:

```javascript
import * as Sentry from '@sentry/browser'

function ErrorBoundaryWithLogging({ children }) {
  const handleError = (error) => {
    // Log to error monitoring service
    Sentry.captureException(error)
    
    // Log to console in development
    if (process.env.NODE_ENV !== 'production') {
      console.error('ErrorBoundary caught:', error)
    }
    
    return (
      <div className="error-fallback">
        <h2>Something went wrong</h2>
        <p>Our team has been notified.</p>
      </div>
    )
  }
  
  return (
    <ErrorBoundary fallback={handleError}>
      {children}
    </ErrorBoundary>
  )
}
```

## Recovery Strategies

### Manual Recovery

Provide a way for users to retry:

```javascript
function RecoverableError({ error, resetError }) {
  return (
    <div>
      <h2>Error: {error.message}</h2>
      <button onClick={resetError}>Try Again</button>
    </div>
  )
}

// Implement reset logic in parent component
function MyComponent() {
  const [key, setKey] = useStore(0)
  
  return (
    <ErrorBoundary
      key={key}
      fallback={(error) => (
        <RecoverableError 
          error={error} 
          resetError={() => setKey(k => k + 1)}
        />
      )}
    >
      <ProblematicComponent />
    </ErrorBoundary>
  )
}
```

Changing the `key` prop forces React to unmount and remount the ErrorBoundary, effectively resetting the error state.

### Automatic Recovery

Implement automatic retry with exponential backoff:

```javascript
function AutoRecoveryBoundary({ children, maxRetries = 3 }) {
  const [retryCount, setRetryCount] = useStore(0)
  const [key, setKey] = useStore(0)
  
  const handleError = (error) => {
    if (retryCount < maxRetries) {
      const delay = Math.pow(2, retryCount) * 1000
      setTimeout(() => {
        setRetryCount(c => c + 1)
        setKey(k => k + 1)
      }, delay)
      
      return <div>Retrying... ({retryCount + 1}/{maxRetries})</div>
    }
    
    return (
      <div>
        <h2>Unable to load component</h2>
        <p>{error.message}</p>
      </div>
    )
  }
  
  return (
    <ErrorBoundary key={key} fallback={handleError}>
      {children}
    </ErrorBoundary>
  )
}
```

## What Error Boundaries Don't Catch

Error boundaries **cannot** catch:

<Warning>
- **Event handlers** - Use try/catch inside handlers
- **Asynchronous code** - Promises, setTimeout, etc.
- **Server-side rendering errors** - SSR uses different error handling
- **Errors in the boundary itself** - The boundary cannot catch its own errors
</Warning>

### Handling Event Handler Errors

Use try/catch for event handlers:

```javascript
function MyComponent() {
  const handleClick = async () => {
    try {
      await riskyOperation()
    } catch (error) {
      console.error('Operation failed:', error)
      // Handle error appropriately
    }
  }
  
  return <button onClick={handleClick}>Click me</button>
}
```

### Handling Async Errors

For async operations, maintain error state:

```javascript
function DataLoader() {
  const [data, setData] = useStore(null)
  const [error, setError] = useStore(null)
  
  useEffect(() => {
    fetchData()
      .then(setData)
      .catch(setError)
  }, [])
  
  if (error) {
    return <div>Error: {error.message}</div>
  }
  
  return <div>{data}</div>
}
```

## Implementation Details

The ErrorBoundary component in RyunixJS is implemented as:

```javascript
import { createElement } from './createElement.js'
import { getState } from '../utils/index.js'

export function ErrorBoundary({ children, fallback }) {
  const state = getState()
  
  // If the engine caught an error for this boundary, render the fallback
  if (state.wipFiber && state.wipFiber.stateError) {
    if (typeof fallback === 'function') {
      return fallback(state.wipFiber.stateError)
    }
    return fallback
  }

  // Otherwise render children normally
  return createElement(
    'ryunix-error-boundary-wrapper', 
    { style: { display: 'contents' } }, 
    children
  )
}

ErrorBoundary.ryunix_type = 'RYUNIX_ERROR_BOUNDARY'
```

Key points:
- Checks `state.wipFiber.stateError` set by the reconciler
- Renders fallback as function or static element
- Uses `display: contents` to avoid affecting layout
- Marked with special type for reconciler detection

## Best Practices

<AccordionGroup>
  <Accordion title="Don't overuse error boundaries" icon="triangle-exclamation">
    Only use error boundaries for truly exceptional cases. Fix bugs instead of catching them in production. Error boundaries are for unpredictable errors, not expected error states.
  </Accordion>
  
  <Accordion title="Provide contextual error messages" icon="message">
    Tailor error messages to the context. A checkout error should suggest different actions than a profile loading error.
  </Accordion>
  
  <Accordion title="Always log errors" icon="file-lines">
    Even with graceful fallback UI, log errors to monitoring services so you can identify and fix issues.
  </Accordion>
  
  <Accordion title="Test error boundaries" icon="vial">
    Explicitly test that error boundaries work as expected by triggering errors in development.
  </Accordion>
  
  <Accordion title="Use multiple boundaries" icon="layer-group">
    Place boundaries at different granularity levels - application, page, and component level.
  </Accordion>
</AccordionGroup>

## Testing Error Boundaries

Test that error boundaries work correctly:

```javascript
function BuggyComponent({ shouldThrow }) {
  if (shouldThrow) {
    throw new Error('Intentional test error')
  }
  return <div>Working correctly</div>
}

function App() {
  const [shouldThrow, setShouldThrow] = useStore(false)
  
  return (
    <div>
      <button onClick={() => setShouldThrow(true)}>
        Trigger Error
      </button>
      
      <ErrorBoundary fallback={<div>Error caught!</div>}>
        <BuggyComponent shouldThrow={shouldThrow} />
      </ErrorBoundary>
    </div>
  )
}
```
